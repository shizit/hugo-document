## JWTとは
JWT（JSON Web Token）は、主に認証・認可（ログイン状態の証明や権限確認）のために使われる  
トークン形式のデータです。
APIやWebアプリで、ユーザーがログインしていることをサーバーが確認するためによく使われます。

## JWTの構造
JWTはヘッダー、ペイロード、シグネチャーの3つのパートで構成されています。

### 1. ヘッダー
ヘッダーでは署名アルゴリズムの種類やトークンの情報を保持します。
``` json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### 2. ペイロード
ペイロードではログインユーザーなどのデータや、トークンの有効期限を保持します。  
ペイロードは暗号化しているわけではなくBase64エンコードのため、  
見られてよいデータを格納します。  
代表的な項目は以下の通りです。
| 項目  | 意味     |
| --- | ------ |
| sub | ユーザーID |
| exp | 有効期限   |
| iat | 発行時間   |
| iss | 発行者    |
``` json
{
  "userId": "123",
  "name": "taro",
  "role": "admin",
  "exp": 1710000000
}
```

### 3. シグネチャー
改ざん防止の署名です。サーバーだけがもつsecret keyで署名します。
``` 
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```
PayloadやHeaderを書き換えると、署名が一致しなくなるためTokenが無効と判定されます。

## JWTを使った認証の流れ
1. ログイン時などのタイミングでサーバーがJWT発行しクライアントで返す
2. サーバーへリクエストする際、HttpヘッダーのAuthorizationなどに1で取得したJWTを設定
3. サーバー側で2のJWTが有効かどうか検証し、レスポンスを返すか判断する

## メリット
サーバー側でログインしているかどうか（=状態）を保持する必要がなくなる（ステートレス）

## デメリット
無効化が難しい（例えばサーバー側でログイン状態を管理している場合、  
ユーザーのログアウト時にサーバー側でログアウト状態にできるが、
クライアント側のJWTでログイン状態を管理している場合、  
もしJWTを盗まれるとユーザーがログアウトしたとしても盗んだJWTでアクセスし続けることができてしまう）  

## デメリットの対策
失効したトークンのリスト（ブラックリスト）をサーバー側で管理したり、  
リフレッシュトークンと組み合わせてアクセストークンの有効期間を極端に短くしたりするなどの  
追加対策が必要になるようです。
